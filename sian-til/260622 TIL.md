## 새로 배운 내용 : LangChain 한 줄 정리

### ▫︎ LangChain 기초

#### LangChain

- 프롬프트, 모델, 파서, 대화 이력, 문서 등의 컴포넌트들을 표준화된 인터페이스로 제공하는 프레임웍
- 각 단계를 컴포넌트로 분리하고 파이프라인으로 연결해준다.
- (사용 이유) 모델을 준비하는 방식이 통일되어 있기 때문에 모델 선택의 자유가 주어진다.
- Agentic framework

#### LCEL(LangChain Expression Language)

- 파이프(`|`) 연산자로 runnable 컴포넌트를 연결하여 체인을 구성하는 핵심 문법 체계
- (사용 방법)
- `prompt | model | parser`
- (사용 이유) 여러 단계를 일관된 방식으로 연결해 코드 구조를 단순화할 수 있고, 기능 변경을 빠르고 안정적으로 할 수 있기 때문

### ▫︎ 프롬프트 구성

#### PromptTemplate

- 변수 자리표시자를 포함한 프롬프트 문자열을 객체로 관리하고 최종 프롬프트를 생성하는 클래스
- (사용 이유) 프롬프트의 틀을 이용해 값만 바꿔서 반복적으로 사용할 수 있게 해준다. 또한 변경 이력 추적이 용이하다.

#### ChatPromptTemplate

- PromptTemplate이 단일 문자열 프롬프트를 만드는 클래스라면 ChatPromptTemplate은 역할이 구분된 메시지 리스트를 입력으로 받아서 메시지 구조도 함께 만들어주는 확장형 템플릿

#### FewShotPromptTemplate

- 입출력 예시를 넘겨주어서 원하는 형식과 패턴의 LLM 응답을 유도하는 템플릿

### ▫︎ 출력 구조화

#### Structured Output

- 정의된 스키마에 맞춰 LLM의 응답을 구조화된 객체로 반환하는 방식
#### JsonOutputParser

- JSON 형태로 파싱하는 출력 파서 클래스
- Structured Output과의 차이점: 구조화를 어디에서 보장하느냐.
    - structured output : 모델 단계에서부터 구조화된 응답을 만들도록 하고
    - jsonoutputparser : 응답 텍스트를 JSON으로 파싱
#### PydanticOutputParser

- Pydantic 모델 인스턴스로 파싱하는 출력 파서 클래스
- 타입 검증과 필드 접근을 제공

### ▫︎ LCEL과 실행 조합

#### Runnable Interface

- 서로 다른 객체를 같은 방식으로 실행하기 위한 표준 메서드 규칙
- 컴포넌트들이 동일한 방식으로 호출되고 파이프로 연결될 수 있게 한다.
    - invoke
    - batch
    - stream
    - ainvoke

### ▫︎ 대화 이력 처리

#### RunnableWithMessageHistory

- 같은 세션ID를 기준으로 대화 이력을 자동으로 불러오고 저장하는 Runnable 래퍼 클래스
- 대화와 대화를 구분할 수 있기 때문에 멀티 유저를 관리할 수 있다.

#### Message Trimming

- 대화 이력이 너무 길면 적당히 줄여서 관리하는 기법
- 최근 이력 개수 또는 토큰 수를 기준으로

### ▫︎ 문서 추상화와 로딩

#### Document

- LangChain에서 문서 한 조각을 표현하는 표준 데이터 객체로 모든 단계에서 이 객체를 사용하여 단계별로 전달한다.
- Document Loader
    - PyPDFLoader : pdf load, metadata: 페이지 번호
    - WebBaseLoader : 웹 페이지 로드, metadata : URL
    - DirectoryLoader : 디렉토리 안에 있는 문서를 한꺼번에 로드, metadata : 파일 경로
