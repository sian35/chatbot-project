## 회고
### 배운 점
- LangChain으로 구현된 RAG 시스템을 LangGraph 기반으로 바꾸는 방법  
선형적이고 순차적으로 실행하는 chain과 달리, 상태 기반의 graph를 이용해 선형적인 구조와 순환하는 구조를 모두 구현할 수 있어서 반복되거나 복잡한 구조를 간편히 구현할 수 있다.
- 로컬 모델과 gemini 모델의 응답 형태 차이  
모델 별로 응답 형태가 다를 수 있기 때문에 정규화 과정을 거쳐서 일관적인 형태로 만들어주어야 한다는 것을 배웠다.


### 어려웠던 점
- 파일의 구조화  
파일 구조에 정답이 없기 때문에 구조화 작업이 오히려 더 어렵게 느껴졌는데 차근 차근 코드를 뜯어보면서 구조화를 진행하니 나만의 구조가 성립되었다. 아직 이 구조가 최적의 상태라고 생각하지 않기 때문에 차차 안정된 구조를 찾아갈 예정이다.
- 프롬프트에 대한 LLM의 이해  
내가 원하는 출력으로 표현하고자 할 때 시스템 프롬프트만으로 지시하는 것은 로컬 모델에게 다소 무리가 있는 작업이어서 직접 아웃풋 형태를 만들어내야 했다. 또한 LLM이 어떤 때에는 프롬프트 지시를 잘 따르는데 어떤 때에는 따르지 못해서 실행할 때마다 결과가 다른 문제가 있었다.  
&rarr; 프롬프트 분리 : 로컬 모델과 프론트 엔드 API 모델의 프롬프트 이해도가 다르기 때문에 프롬프트를 모델에 따라 분리하여 구현함.

### 느낀 점
- RAG 시스템을 바닐라 RAG &rarr; LangChain &rarr; LangGraph로 발전시켜가며 RAG에 대해 이해할 수 있게 되었다.
- 기능적으로 현재 구현한 시스템이 너무 단순하다는 것을 알고 있기에 발전시킬 방향을 빠른 시일 내에 정해서 기능을 확장해야 한다.

### 개선 방향
- 평가 데이터 관리  
평가 데이터가 많아지면 데이터 파일(JSON/YAML)로 분리하기 또는 LangSmith Dataset으로 관리
- 기능 발전시키기  
단순히 내가 작성한 문서에서의 검색이 아니라 더 다양한 소스에서 검색이 가능하도록 발전시키고 싶다.  
특히 공식 문서에서 원하는 정보를 가져오기 등 개발 공부를 도와주는 기능을 확장하고 싶다.   
또한 ReAct 패턴을 적용해서 보다 다양한 도구들을 사용하도록 개선하여 다양한 기능을 LLM이 수행할 수 있도록 하고 싶다.
- 좀 더 특화된 주제 찾기  
현재는 가장 구하기 쉬운 문서인 내가 작성한 TIL(Today I Learned) 파일들을 검색하는 시스템으로 구현되어 있는데 더 실용적인 주제를 찾아서 유용한 RAG 시스템으로 만들고 싶다.
